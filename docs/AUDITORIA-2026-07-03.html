Relatório de auditoria técnica

Helper Financeiro v2 — pontos fortes, fracos e roadmap

Análise independente do código, da stack e da aderência às metodologias Spec-Driven Development e Harness-Driven Development declaradas pelo próprio projeto.

Data: 03/07/2026 Escopo: C:\helper_financeiro (~2.400 linhas Python, 9 docs SDD, 3 ADRs) Verificação: suíte executada — 20/20 testes verdes em 0,20s, offline
1 · Sumário executivo

Um esqueleto excepcional — mas ainda um esqueleto

O Helper Financeiro é um app desktop (tkinter) de diagnóstico de dívidas que gera planilha, relatório e cartas de negociação, com uma camada v2 de IA ("CONSELHEIRO") governada por guardrails. A qualidade da governança é rara de se ver em projetos desse porte: constituição com princípios invioláveis, SPEC em sintaxe EARS com REQ-IDs, plano de arquitetura, tasks rastreáveis e um harness de testes que realmente roda offline.

A opinião honesta: hoje o produto que o usuário final recebe é a v1 determinística. A camada de IA — o argumento central da v2 — não funciona de ponta a ponta: os providers reais (OllamaProvider, OpenAICompatProvider) são stubs que lançam NotImplementedError, e a GUI não importa nada de agent/. Além disso, o projeto não é um repositório git e não tem CI — o que contradiz a própria cultura de rastreabilidade que os documentos pregam. O alicerce merece nota alta; a casa está com um andar construído de quatro.

3 · Pontos fortes

O que o projeto acerta — e acerta muito

Separação determinístico × LLM como princípio de arquiteturaDiferencial

P1 ("o LLM nunca calcula; números vêm do core/") é a decisão mais importante do projeto e está implementada de verdade: agent/agente.py monta FatosFinanceiros a partir do core e o validador numérico rejeita qualquer cifra órfã na saída do LLM. É o padrão correto para IA em domínio financeiro, e pouquíssimos projetos o aplicam com essa disciplina.

core/calculos.py · agent/agente.py:32-75 · guardrails/validador_numerico.py

Degradação segura (fail-safe) implementada e testada

Se o provider falha, o schema não adere ou um guardrail dispara, o sistema entrega o diagnóstico determinístico completo com o motivo registrado — o usuário nunca fica sem resultado por causa do LLM. Coberto por test_degradacao.py.

agent/agente.py:78-126 · tests/test_degradacao.py

Guardrails concretos, não decorativos

Grounding numérico com tolerância ±1% e múltiplas interpretações pt-BR/US; anonimização de credores com mapa só em memória (LGPD); filtro de conteúdo proibido (recomendação de investimento, promessa de retorno); aviso legal garantido. Cada um tem teste dedicado — a regra "guardrail sem teste é inexistente" (P6) é cumprida no M1.

guardrails/ (3 módulos) · tests/test_grounding.py · test_pii.py · test_conteudo.py · test_injecao.py

Documentação SDD exemplar

SPEC em EARS com REQ-IDs prefixados por categoria, matriz REQ→task→teste, constituição versionada com processo de emenda, ADRs para decisões estruturais e AGENTS.md orientando agentes de código (incluindo o protocolo NEEDS_CLARIFICATION anti-improviso). Serve de gabarito para outros projetos.

docs/SPEC.md · docs/CONSTITUTION.md · docs/adr/ · AGENTS.md

Core matematicamente sólido e legível

Price, saldo devedor, CET por bisseção (com justificativa de convergência), simulador avalanche/bola de neve com trava de 600 meses e cascata de sobras correta. Comentários pedagógicos de alta qualidade explicam o porquê, não o óbvio.

core/calculos.py:50-78 · core/estrategias.py:30-80

Local-first por padrão

Config aponta para Ollama local; tudo roda offline; PDF é tratado como entrada não-confiável (P5) e nunca vai cru ao LLM — só fatos tipados trafegam. Postura correta de privacidade para dados financeiros pessoais.

agent/config.py · docs/ADR-0002 · tests/test_injecao.py

4 · Pontos fracos

Achados, por severidade

F-01 · O projeto não é um repositório gitCrítico

Não há .git, .gitignore nem histórico. Isso invalida na prática vários compromissos dos próprios docs: "todo commit cita o REQ-ID" (AGENTS.md), a ata de freeze com SHA-256 (FREEZE.md) e qualquer noção de auditoria de mudanças. Sem controle de versão, todo o edifício de rastreabilidade repousa sobre nada.

git status → "not a git repository"

F-02 · A proposta de valor da v2 não chega ao usuárioCrítico

Os dois providers reais são stubs (NotImplementedError) e a GUI não referencia agent/ em nenhuma linha — o CONSELHEIRO só existe em demo_agente.py com FakeProvider. O TASKS.md é transparente sobre isso (M2/M3 pendentes), mas o README vende a v2 no presente do indicativo. Risco de "demo-ware": pipeline validado que nunca vira produto.

agent/provider.py:93,103 · grep "agent" em gui/app.py → 0 resultados

F-03 · Zero automação de qualidade: sem CI, lint, type-check ou coberturaCrítico

Não existe pyproject.toml, ruff, mypy, pre-commit nem workflow de CI. Os "Gates A/B/C" do HARNESS.md são executados apenas na honra, manualmente. Um harness que não roda sozinho a cada mudança não é um portão — é uma placa de aviso.

ls raiz → apenas requirements*.txt; nenhum arquivo de config de ferramenta

F-04 · Gate B (planilhas sem erro de fórmula) não tem teste — violação do próprio P6Alto

A constituição diz que "guardrail sem teste é inexistente" e o HARNESS declara o Gate B como bloqueante ("recálculo + harness"), mas outputs/ (550 linhas: planilha, relatório, proposta) tem zero testes. Pelo critério do projeto, H4 não existe.

tests/ cobre core+agent+guardrails; nenhum test_planilha/test_relatorio · docs/CONSTITUTION.md H4

F-05 · Dependência circular guardrails ↔ agent viola a regra de camadasAlto

O PLAN manda "setas só apontam para baixo", mas guardrails/validador_numerico.py e conteudo.py importam agent.schemas, enquanto agent/agente.py importa guardrails. Funciona porque agent/__init__.py é vazio, mas é acoplamento mútuo. Os schemas são contratos — devem morar numa camada própria (ex.: contracts/) da qual ambos dependem.

guardrails/validador_numerico.py:14 · guardrails/conteudo.py:12 · agent/agente.py:24-27 · docs/PLAN.md §1

F-06 · REQ-LLM-002 promete retry que não existeAlto

O SPEC exige "tentar 1 (uma) recuperação" quando a saída não adere ao schema; agente.py degrada direto, sem retry. Divergência silenciosa entre spec e código — exatamente o que o SDD existe para impedir. Ou implementa-se o retry, ou emenda-se o SPEC.

docs/SPEC.md REQ-LLM-002 · agent/agente.py:105-107

F-07 · Checagem final de PII não está ligada ao pipelineAlto

contem_pii() existe e é testada, mas nenhum ponto do fluxo a chama antes de uma futura chamada cloud; desanonimizar() também está órfã. Quando o T-202 for implementado, nada força a checagem — o guardrail H2 hoje protege por construção (só fatos tipados saem), mas o cinto de segurança final está desconectado.

guardrails/pii.py:43-62 · grep contem_pii em agent/ → 0 chamadas

F-08 · GUI monolítica, sem threading e com formatação BR frágilMédio

gui/app.py concentra 495 linhas (janela + estado + validação). Chamadas de LLM (30–60s no Ollama) congelariam a interface — o T-303 reconhece, mas o design atual não prepara o terreno (nenhum controller separado). Nome e CPF são coletados na aba Perfil mas nunca usados — colete apenas quando a carta precisar (minimização de dados, LGPD).

gui/app.py:39-120 · TASKS T-302/T-303

F-09 · Grounding numérico é heurístico e tem falsos-limites conhecidosMédio

Inteiros ≤3 são isentos (um LLM que diga "3 vezes mais caro" passa); anos ("2026") e números compostos legítimos seriam órfãos (falso positivo → degrada sem necessidade); o conjunto permitido inclui ×100 mas não ÷100. É um bom guardrail de segurança, mas trate-o como rede, não como prova — e documente os limites no HARNESS.

guardrails/validador_numerico.py:120-127

F-10 · Simulador diverge do cronograma Price sem documentarMédio

simular_quitacao() aplica juros compostos sobre o saldo e paga a parcela nominal constante — um modelo simplificado que não coincide com a evolução real de um contrato Price (onde a parcela embute a amortização programada). Para comparação relativa entre estratégias serve, mas os "meses até quitar" absolutos podem divergir do contrato real. Isso deveria estar declarado no SPEC e no relatório gerado.

core/estrategias.py:47-78

F-11 · Config lê variáveis de ambiente na importação, não na execuçãoBaixo

Os defaults de ConfigAgente chamam os.getenv() no corpo da classe — avaliados uma única vez no import. Mudar HF_PROVIDER depois de importar o módulo não tem efeito; em testes isso gera acoplamento à ordem de import. Mova a leitura para carregar_config() ou use field(default_factory=...).

agent/config.py:13-21

F-12 · Diretório harness/ vazio e testes do core rasosBaixo

harness/ contém só um __init__.py — remova ou dê função. test_core.py tem 40 linhas para 600 linhas de matemática financeira; casos de borda (taxa 0, n=1, saldo ínfimo, dívidas não-quitáveis) merecem testes baseados em propriedades (Hypothesis: ex. saldo_devedor_price(pv,i,n,n) == 0 para todo pv,i,n).

harness/__init__.py · tests/test_core.py

6 · Metodologia

Spec-Driven Development e Harness-Driven Development na prática

O projeto não precisa ser convencido dessas metodologias — ele já as adota nominalmente. A auditoria avaliou a aderência real e onde apertar o parafuso.

Onde o SDD está funcionando

  • Cadeia de artefatos completa: PRD → SPEC (EARS) → PLAN → TASKS → código, com CONSTITUTION acima de tudo e ADRs para desvios. É o fluxo canônico (mesmo espírito do spec-kit/OpenSpec).
  • Requisitos testáveis: EARS com gatilhos e respostas verificáveis; cada REQ-GRD tem teste nomeado na matriz.
  • Anti-improviso: P7 + NEEDS_CLARIFICATION + denylist controlam o maior risco de desenvolvimento com agentes de código: a invenção de escopo.

Onde o SDD está furado

  • Spec drift já começou (F-06: retry prometido e ausente). Melhor prática: um teste de conformidade por REQ, e revisão de SPEC como parte do DoD de cada task — a spec é contrato vivo, não prosa de abertura.
  • Rastreabilidade sem git é ficção (F-01): "todo commit cita o REQ-ID" exige commits. O primeiro ato de SDD real deste projeto é git init.
  • Status ✅ auto-declarado: TASKS marca M1 como feito pelo próprio scaffold. Melhor prática: o status vem do CI (teste verde no pipeline), não do documento.

Onde o harness está funcionando

  • Determinístico e offline por design: FakeProvider que respeita H1 por construção permite CI rápido e sem flakiness — exatamente o padrão recomendado para pipelines com LLM.
  • Testes adversariais de verdade: fixtures com número fabricado, recomendação de investimento e PDF malicioso (prompt injection) cobrem os modos de falha que importam.

Onde o harness precisa crescer

  • Gates que não executam sozinhos não são gates (F-03): a essência do harness-driven é o loop automático — cada mudança passa pela bancada sem depender de disciplina humana.
  • Cobertura desigual: outputs/ sem teste (F-04), core raso (F-12), GUI sem sequer smoke test de importação.
  • Nível 2 do harness ausente e necessário para o M2: quando o Ollama real entrar, o FakeProvider deixa de cobrir o risco novo (aderência de schema por modelo pequeno). Planejar: suíte de integração marcada (-m ollama) com casos-ouro, métrica de taxa de aderência ao schema, e o LLM-as-judge do HARNESS §4 implementado com rubrica versionada.
  • Golden files para outputs: planilha/docx gerados a partir dos perfis-ouro, com asserções estruturais (fórmulas recalculam, aviso legal presente, seção IA separada) — cobre H3/H4 de uma vez.
7 · Melhorias

Estruturais e de produto

Estruturais (engenharia)

  • Camada contracts/: mover agent/schemas.py para um pacote de contratos sem dependências; agent e guardrails passam a depender só dele (resolve F-05 e deixa o diagrama do PLAN verdadeiro).
  • Pipeline de guardrails declarativo: hoje a orquestração em agente.py é if-encadeado; uma lista ordenada de checagens (nome, fn, é_crítica) torna trivial adicionar guardrail novo com log e teste padronizados.
  • Separar apresentação de estado na GUI: extrair um AppController testável; o LLM roda em threading.Thread com fila para o mainloop (prepara T-302/T-303 sem congelar a UI).
  • Logging estruturado: configurar logging com arquivo local rotativo (sem PII, conforme REQ-SEC-001) — hoje os warnings de degradação vão para lugar nenhum no .exe.
  • Validação nos modelos do core: Divida(saldo_devedor=-100) passa silenciosamente hoje; validadores pydantic (ou __post_init__) param dado ruim na porta.

Produto

  • Persistência de perfil (a maior dor de uso hoje): o usuário redigita tudo a cada sessão. Salvar/abrir perfil em JSON local — com aviso claro de onde fica o arquivo — muda o produto de "calculadora" para "acompanhamento". Candidato a REQ-F-006 via processo de emenda.
  • Acompanhamento mensal: com persistência, comparar mês a mês (saldo caindo, comprometimento melhorando) cria o loop de motivação que métodos como bola de neve exploram.
  • Amortização SAC: financiamentos imobiliários (caso comum de portabilidade) usam SAC; hoje só Price é modelado.
  • Transparência do modelo: declarar no relatório a simplificação do simulador (F-10) e a data-base das faixas de referência (4% a.m., limites 30/50%).
  • OCR local opcional para contratos escaneados — o extrator já detecta o caso e avisa; falta o caminho feliz.
  • CLI mínima (python -m helper_financeiro analisar perfil.json): barata graças à arquitetura em camadas, ótima para o harness e para usuários avançados.
  • Distribuição confiável: .exe não assinado dispara SmartScreen; documentar o aviso e publicar hash na página de release reduz abandono na instalação.
8 · Roadmap

Plano de implementação em 5 fases

Esforço estimado para 1 dev em tempo parcial. As fases 0 e 1 são pré-requisito de tudo; a ordem 2→3→4 segue os milestones M2→M3→M4 já planejados pelo projeto.

Fase 0 · Fundação — 2 a 3 dias

Transformar a pasta num projeto de engenharia

  • git init + .gitignore + primeiro commit citando esta auditoria (fecha F-01).
  • pyproject.toml com uv; migrar requirements para dependências + grupo dev.
  • ruff + mypy + pre-commit; corrigir o que apontarem (código já tipado — custo baixo).
  • CI (GitHub Actions): lint → mypy → pytest -q + pytest-cov com piso inicial de 70% (fecha F-03).
  • Remover harness/ vazio ou movê-lo para dentro de tests/ (F-12).
Fase 1 · Dívida de conformidade — 1 semana

Fazer os docs e o código dizerem a mesma coisa

  • Criar contracts/ e quebrar o ciclo guardrails↔agent (F-05); atualizar PLAN §1.
  • Implementar o retry único do REQ-LLM-002 no orquestrador — ou emendar o SPEC via ADR (F-06).
  • Testes de outputs/: golden files dos 3 perfis-ouro + recálculo de fórmulas → Gate B real no CI (F-04).
  • Ligar contem_pii() como checagem final obrigatória no caminho cloud e testar (F-07).
  • Corrigir leitura de env em config.py (F-11); testes de propriedade com Hypothesis no core (F-12).
  • Documentar limites do grounding (F-09) e a simplificação do simulador (F-10) no HARNESS/SPEC.
Fase 2 · M2, providers reais — 2 a 3 semanas

O CONSELHEIRO deixa de ser fake

  • T-201 OllamaProvider com instructor (ou PydanticAI — decidir por ADR) + timeout + retry.
  • Bancada de aderência de schema: medir taxa de sucesso por modelo (7B vs 14B) com os casos-ouro; escolher o mínimo viável.
  • T-202 OpenAICompatProvider com contem_pii() obrigatório pré-envio.
  • T-205 cache local de análise; T-206 teste de degradação com Ollama desligado de verdade.
  • Suíte de integração pytest -m ollama (fora do gate bloqueante) + LLM-as-judge com rubrica do HARNESS §4.
Fase 3 · M3, integração de saída — 2 semanas

A IA chega ao usuário — visivelmente separada dos números

  • Extrair AppController; LLM em thread com barra de progresso (T-303) e indicador de modo degradado (T-304).
  • Painel "Análise do CONSELHEIRO (assistida por IA)" na GUI com desanonimização para exibição local (T-302).
  • Seção IA no .docx, visualmente separada do determinístico, com aviso legal (T-301).
  • Produto: salvar/abrir perfil em JSON local (novo REQ-F-006 via emenda) — destrava acompanhamento mensal na v2.1.
Fase 4 · M4, empacotamento e release — 1 semana

Entregar com confiança

  • Build PyInstaller no CI com pydantic/instructor incluídos (T-401); smoke test do .exe.
  • Revisão de segurança: varredura de PII/keys em logs e artefatos (T-403).
  • FREEZE.md alimentado automaticamente com SHA-256 do artefato (T-402); release versionada com hash publicado.
  • Backlog v2.1+: SAC, OCR local, CLI, comparativo mensal.
9 · Conclusão

Veredito

Este é um dos scaffolds SDD mais bem pensados que se encontra em projetos individuais: a constituição, os guardrails testados e a decisão de manter o LLM fora da matemática são exatamente as escolhas certas para IA aplicada a finanças pessoais. O risco do projeto não é técnico — é de execução e honestidade de status: sem git e sem CI, a rastreabilidade prometida não existe na prática; sem os milestones M2/M3, a v2 é uma promessa bem documentada, e o spec drift já apareceu (F-06) antes mesmo do primeiro provider real.

A boa notícia: as fases 0 e 1 custam cerca de duas semanas e não exigem nenhuma decisão difícil — só transformam em infraestrutura o que hoje é disciplina declarada. Feito isso, o projeto colhe o que o próprio desenho já plantou.
